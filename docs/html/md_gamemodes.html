Gamemode Anatomy

To host your own server in Maker: Online, you'll need to create a gamemode DLL which references the Maker.API DLL. There might also be a need to reference the Basick DLL, which contains some common types used within the API and the core Maker: Online code.

The gamemode DLL you build will then have to be referenced within the gamemode-manifest.xml file.

Maker Event Functions

Maker Event Functions are equivalent to Unity Event Functions in Unity. Maker: Online is event-driven, thus features can be easily added when different events occur. Different Maker Event Functions occur depending on which base class is inherited.

Maker Event Functions should be declared non-public to prevent external sources other than Maker: Online invoking them. When inheriting from base classes, you'll notice that there's no override for most event functions, this is because there's no implementation, this is for various reasons...

  • To support dependency injection.
  • For optimisation within Maker: Online.

Thus, you won't see the non-implemented Maker Event Functions documented within the official API documentation. You will, however, see all of the other properties and methods documented, alongside the few Maker Event Functions that do happen to have a base implementation and are virtual.

Note
In such instances when an event function is virtual, it's always advised to call its base method when overriding.
Virtual methods are marked with a dashed border in the Order of Execution diagrams below.

Base Classes

Base classes in Maker.API are somewhat equivalent to MonoBehaviour's in Unity. Base classes provide default behaviour, but should ideally be inherited from in order to extend functionality.

Below, each base class exposed by Maker.API is explained along with their Maker Event Functions.

Gamemode

See Maker.API.Gamemode.

The abstract base gamemode class is the root of any type that controls the overall functioning of the hosted gamemode. This is the primary starting point when building your gamemode. There can be any number of gamemode classes, and at least one gamemode class must be declared. Plugins will usually provide their own gamemode class so that they can perform their own setup. Plugin gamemodes are always initialised before the core gamemode implementations. A single instance of each gamemode type will be instantiated once the DLL's are loaded and the server is about to start accepting connections.

This is the only abstract base class, and it must be inherited from as there's no default implementation.

Maker Event Functions

OnLoad

‍ Called when the gamemode is loaded and initializing. This is the ideal place to perform some initial setup.

OnServerStart

‍ Called when the gamemode DLL has been fully loaded and parsed, all types cached, and the server is ready to begin accepting connections. Any final setup should be performed here.

OnServerStop

‍ Called when the server has cut all connections and stopped, this happens right before the program exits. Clean up should be performed here.

OnUnload

‍ Called when the gamemode is unloaded and the program exits. This is the ideal place to perform any final clean up.

Player

See Maker.API.Player.

A base class which represents a Maker: Online player. This base class is used by default when a player joins the server, until a different type is specified. It's implementation is minimal, so if this default type is used, nothing will happen with the player once they have connected. This type is only here to allow players to connect even when there isn't a proper implementation of player. ideal for your initial testing when you have declared a gamemode class and nothing else.

Maker Event Functions

OnInit

‍ This virtual event function allows Maker: Online to initialise the Player. This event can be overridden, but it's advised to put all code after the base call, to ensure that the player is fully initialised.

OnConnect

‍ Called when a player connects to the server and after the player has been fully initialised.

OnDisconnect

‍ Called when a player disconnects from the server. This is the ideal place to perform clean up.
